# Response Strategy Engine — Differentiated Immune Response for LLM Security

> Don't just detect — react tactically. Five response types
> based on biological immune-response patterns.
>
> See [`outputs/technical_report_2026-04-07.md`](../outputs/technical_report_2026-04-07.md)
> Section 3.4 for the architectural context.

## The five response types

LLD does not have a single "block / allow" decision. Instead it
selects one of five response strategies based on (a) the
confidence of the detection, (b) the session history, and
(c) the attack type.

| Strategy | Biological analogue | Behaviour | When |
|---|---|---|---|
| **TOLERATE** | Anergy / non-response | Allow the request, log for analysis | Confidence < 0.3, no prior incidents |
| **SANDBOX** | Encapsulation (granuloma) | Allow but in a reduced-capability context | 0.3 ≤ confidence < 0.5 |
| **INFLAME** | Inflammatory response | Block + raise alert + tighten thresholds | 0.5 ≤ confidence < 0.7 |
| **DECEIVE** | Mimicry / camouflage | Return a fake (honeypot) response with watermarks | 0.7 ≤ confidence < 0.9, attack appears tactical |
| **TERMINATE** | Apoptosis | Block + ban session for N seconds | confidence ≥ 0.9 OR repeat offender |

## Decision matrix

```
                   confidence < 0.3       0.3 - 0.5     0.5 - 0.7     0.7 - 0.9        >= 0.9
                   ────────────────       ─────────     ─────────     ─────────        ──────
session: clean     TOLERATE              SANDBOX        INFLAME       DECEIVE          TERMINATE
1 prior incident   SANDBOX               INFLAME        INFLAME       DECEIVE          TERMINATE
2+ prior incidents INFLAME               DECEIVE        DECEIVE       TERMINATE        TERMINATE
attack repeat       DECEIVE              DECEIVE        TERMINATE     TERMINATE        TERMINATE
```

The matrix is implemented in
[`lld/response_strategy.py`](../lld/response_strategy.py) by the
`StrategySelector` class.

## Honeypot responses (DECEIVE)

When the strategy is DECEIVE, LLD does not return an error or a
blocked-message. Instead it returns a *plausible-looking* fake
response generated by `HoneypotGenerator`. This serves three
purposes:

1. **Information starvation.** The attacker receives no signal
   that they have been detected, so they cannot adapt their
   approach.
2. **Time waste.** The attacker may spend cycles processing the
   fake response, slowing their iteration loop.
3. **Attribution.** The fake response carries a steganographic
   watermark (zero-width characters, homoglyphs, or canary URLs)
   that lets the operator trace where the response was published
   if it appears in dumps, leak sites, or downstream systems.

See [`lld/watermark.py`](../lld/watermark.py) for the watermarking
implementation.

## Sandbox responses (SANDBOX)

When the strategy is SANDBOX, LLD allows the request but reduces
the response capability:

- Output length capped at a small number of tokens
- Code execution disabled
- Access to tool-use APIs revoked
- Conservative system prompt substituted

This is useful for borderline cases where blocking would be too
aggressive (and risk false-positive complaints) but full
trust is not appropriate either.

See [`lld/response_strategy.py`](../lld/response_strategy.py)
for the `SandboxResponse` class.

## Game-theoretic motivation

The five-type response engine is motivated by a simple game-
theoretic argument: an attacker who receives a binary
allow / block signal can use that signal to learn (probe-and-adapt
attacks). An attacker who receives a multi-modal response —
sometimes blocked, sometimes sandboxed, sometimes deceived —
cannot reliably distinguish between "my attack worked" and
"the defense is messing with me."

This is the LLM analogue of the OODA-loop disruption principle:
**poison the attacker's observations, not just their actions.**
